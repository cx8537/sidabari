모든 코드 작성, 수정, 디버깅 작업 전에 이 문서를 읽고 시작하세요.

## 1. 절대 원칙 (NEVER 규칙)

다음 규칙은 **어떠한 경우에도** 위반해서는 안 됩니다.

### 1.1 코드를 추측하지 말 것

> **추측은 모든 버그의 시작이다.**

- ❌ 절대 금지:
  - 함수가 어떻게 동작할 거라고 **상상해서** 수정하기
  - 파일을 읽지 않고 "아마 이럴 것이다"라고 가정한 채 코드 작성
  - 에러 메시지 일부만 보고 원인을 단정한 뒤 즉흥적으로 패치
  - 라이브러리 API를 기억에 의존해서 호출 (특히 Rust crate, Tauri API, xterm.js API)
  - 본 적 없는 파일을 "이런 구조일 것이다"라고 가정

- ✅ 반드시 할 것:
  - **수정 전에 관련 코드를 먼저 읽는다.** 함수 본체, 호출부, 인터페이스, 타입 정의 모두.
  - 외부 라이브러리 사용 시 **공식 문서 또는 실제 소스를 확인**한다.
  - 에러 발생 시 **스택 트레이스 전체와 Caused by 체인을 끝까지 추적**한다.
  - 가설을 세웠으면 코드를 봐서 **검증**한 뒤에야 수정한다.
  - 모르는 코드는 "모른다"고 명시하고, 확인 후 진행한다.

### 1.2 보안을 최우선으로

> **이 도구는 SSH 접속, SFTP 업로드, EC2 명령 실행을 다룬다. 보안 사고는 즉시 실서비스 사고로 이어진다.**

#### 1.2.1 자격증명/키 파일 처리

- **개인키(.pem) 파일 내용을 절대 코드/로그/설정 파일/메모리 외 영역에 저장하지 않는다.**
  - 설정에는 **파일 경로만** 저장한다.
  - 메모리에 로드한 키는 사용 후 즉시 폐기 (가능한 경우 `zeroize` 등 활용).
- **로그/콘솔 출력에 자격증명을 절대 찍지 않는다.**
  - SSH 비밀번호, 키 내용, JWT, API 키, 토큰 등.
  - 디버그 로그도 마찬가지. `tracing::debug!`로도 안 됨.
- **자격증명을 화면에 표시할 때는 마스킹**한다 (예: `****1234`).
- **에러 메시지에 자격증명이 포함되지 않도록 필터링**한다.

#### 1.2.2 명령 실행

- **사용자 입력을 셸 명령으로 직접 연결하지 않는다.** 
  - ❌ 금지: `format!("ssh {}@{}", user, host)` 같은 문자열 조합 후 셸 실행
  - ✅ 권장: SSH 라이브러리(`russh`)의 구조화된 API 사용
- **빌드 명령, 진단 명령 등 사용자 정의 명령은 신뢰 경계로 간주**한다.
  - 사용자가 의도적으로 입력했다고 가정 (사용자 자신이 자기 PC에서 실행)
  - 다만 **자동 실행 전에 항상 사용자 승인을 거친다** (사양서 참조).
- **셸 인젝션 방지**: 외부에서 받은 텍스트 (예: 로그 메시지, Claude 응답)를 **절대 명령으로 실행하지 않는다.**
  - 진단 루프에서 Claude가 추천한 명령도 **사용자가 명시적으로 클릭해야 실행**된다.

#### 1.2.3 SSH/SFTP 연결

- **호스트 키 검증을 활성화**한다. `known_hosts`를 확인하고, MITM 방지.
  - 첫 접속 시 호스트 키를 사용자에게 보여주고 명시적 승인을 받는다 (TOFU 패턴).
  - 자동 수락 모드 금지.
- **SSH 연결 타임아웃을 명시적으로 설정**한다 (사양서: 10초 grace).
- **연결 실패 시 자동 재시도하지 않는다** (사양서 정책).

#### 1.2.4 파일 시스템

- **사용자 홈 디렉토리 외부 파일에 함부로 접근하지 않는다.**
- **임의 경로 쓰기 금지**: 설정에서 경로를 받을 때 정규화하고, 의도된 디렉토리 내인지 검증.
- **Tauri의 fs scope를 좁게 설정**한다. 필요한 경로만 화이트리스트.
- **path traversal 방어**: `..`, 절대경로, 심볼릭 링크 우회 등 검증.

#### 1.2.5 IPC (Tauri Command)

- **Rust → JS로 노출하는 Tauri command는 모두 검증**한다.
  - 입력값 길이 제한, 형식 검증, 금지 문자 검사.
- **민감한 동작에는 사용자 확인을 강제**한다 (예: SSH 키 변경, 명령 실행).
- **command 권한 범위를 최소화**한다. 한 command가 너무 많은 일을 하지 않도록.

#### 1.2.6 의존성 관리

- **`cargo audit`을 정기적으로 실행**해 알려진 취약점 확인.
- **`npm audit`도 마찬가지.**
- **새 의존성 추가 시 메인테이너, 다운로드 수, 최근 업데이트를 확인**한다. 무명 패키지 주의.

#### 1.2.7 로그 영구화

- **이벤트 로그를 SQLite에 저장할 때, 자격증명/민감 정보가 들어가지 않도록 필터링.**
- 로그 파일 권한을 `0600` (사용자만 읽기/쓰기)으로 설정.

### 1.3 자동화 안전 규칙

- **모든 자동 재시도는 금지.** 실패 시 멈추고 사용자가 결정.
- **Claude의 추천을 자동 실행하지 않는다.** 항상 사용자 승인 후 실행.
- **강제 중단 시 SSH 채널을 닫지 않는다.** 사용자가 같은 세션에서 수작업 이어받도록.
- **워크플로우 단계 간 자동 진행은 사양서에 명시된 경우만 허용.**

## 2. 작업 절차

### 2.1 새 기능 구현 시

1. **사양서(SIDABARI_SPEC.md) 해당 부분을 다시 읽는다.**
2. 영향받는 기존 코드를 모두 찾아 읽는다 (`grep`, `view`, 호출부 추적).
3. 데이터 흐름과 책임 경계를 파악한다.
4. 보안 영향을 검토한다 (위 1.2 항목 적용 여부 점검).
5. 작은 단위로 구현하고, 각 단위를 검증한다.
6. 변경 사항이 사양서와 일치하는지 확인한다.

### 2.2 버그 수정 시

> **이 절에서 추측 금지 원칙이 가장 강하게 적용된다.**

1. **에러 재현 조건을 명확히 한다.** 어떤 입력에서, 어떤 환경에서.
2. **스택 트레이스를 끝까지 읽는다.** 표면 예외만 보지 말고, **`Caused by` 체인을 끝까지 추적**한다.
   - JVM 예외든 Rust panic이든 JS 에러든 동일.
   - 진짜 원인은 마지막 `Caused by`에 있는 경우가 대부분.
3. **관련 코드를 직접 본다.** 추측하지 말고:
   - 에러가 발생한 함수 본체
   - 호출부 (어떤 인자로 부르는지)
   - 인자의 타입 정의
   - 관련 라이브러리 함수의 실제 동작 (필요 시 의존성 소스 열어보기)
4. **가설을 명시적으로 세운다.** "X 때문에 발생한다"고 단정하지 말고, "X가 원인일 가능성이 있다"고 표현.
5. **가설을 검증한다.** 코드를 보고 그 가설이 가능한지, 다른 가능성은 없는지.
6. **재현되는 최소 케이스를 만든다.** 가능하면.
7. **수정한다.** 단, 같은 패턴의 다른 곳도 함께 점검.
8. **수정 후 같은 케이스가 재현되지 않는지 확인.**

### 2.3 의심스러울 때 행동

다음 상황에서는 **반드시 멈추고 확인**한다:

- 코드를 보지 않은 함수를 호출하려 할 때 → 본다
- "아마 이렇게 동작할 것 같다"는 생각이 들 때 → 확인한다
- 에러 메시지를 보고 즉시 패치가 떠오를 때 → 의심한다, 진짜 원인인지 확인
- 라이브러리 API 시그니처를 기억에 의존해 쓰려 할 때 → 문서/소스 확인
- 보안에 영향을 줄 수 있는 변경을 하려 할 때 → 1.2 절을 다시 읽는다
- 사용자 자격증명이나 키 파일 관련 코드를 작성할 때 → 1.2.1을 다시 읽는다

## 3. 코드 스타일

### 3.1 Rust

- `clippy` 경고는 무시하지 않는다. 끄려면 명시적 이유 주석.
- `unwrap()`, `expect()`는 **정말 panic이 정상 동작인 경우에만** 사용. 그 외엔 `Result` 반환.
- 에러는 `thiserror` 등으로 타입화. `Box<dyn Error>` 남용 금지.
- async 함수는 명시적 타임아웃을 가진다 (특히 네트워크/SSH).
- `unsafe` 블록은 **반드시 안전성 근거를 주석으로** 남긴다.

### 3.2 TypeScript

- `any` 사용 금지. 정 필요하면 `unknown` 후 narrowing.
- 외부 입력(IPC 응답 포함)은 런타임 검증 (zod 등).
- React 컴포넌트는 단일 책임 유지. 패널 단위 모듈화 (사양서 4.3).
- 상태는 Zustand에 두되, 컴포넌트 로컬 상태로 충분한 건 로컬에.

### 3.3 공통

- 함수 길이가 길어지면 분리. 50줄 넘으면 의심.
- 매직 넘버/문자열은 상수로 추출.
- 주석은 "왜"를 설명. "무엇"은 코드가 말하게.

- "확실하지 않습니다"라고 말하는 것이 잘못된 답을 주는 것보다 낫다.
